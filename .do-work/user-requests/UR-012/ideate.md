# Ideate — UR-012

**Reviewed:** 2026-07-09

## Explorer — Assumptions & Perspectives

- **The brief assumes every "CONFIRMED" finding is a bug to fix, but at least one is a deliberate design.** Finding #1 recommends sharding read committed-only timings (`readMerged()`). UR-011/REQ-051 ("Read-only load — consumers overlay pending in memory; explicit `mergeToDisk()`") and the standing decision `2026-07-09 | UR-011 | warp shard/timings are read-only (in-memory pending overlay); disk merge only via explicit warp merge` establish the pending-overlay behaviour on purpose, chosen at a question gate because "read-only CI artifact restores must work." Acting on #1 as written would reverse a decision made three archived REQs ago.
- **The affected subsystem was already code-reviewed and hardened by UR-011.** REQ-047 through REQ-052 fixed a prior review of this exact s3 timing/sharding code. Several of these new findings land on files those REQs just rewrote — so the question for each is "residual gap or intended behaviour?", not "new bug."
- **"Green CI with silently-skipped tests" (finding #1) is the alarming claim, but its trigger — non-identical pending state across shard machines — is undefined in the brief.** In the intended CI flow each shard machine restores the same committed `timings.json` artifact and has no local `pending/` at shard time (pending is written *during* a run, not before one). The divergence scenario needs a concrete path where two machines legitimately hold different pending sets at shard time; without one, #1 is theoretical.

## Challenger — Risks & Edge Cases

- **Finding #2 (SnapshotStore unimported `RuntimeException`) is a genuine regression introduced by UR-011's own REQ-047.** REQ-047 ("Extract shared file-lock helper from SnapshotStore") removed `use RuntimeException;` when it swapped in `use ...FileLock;` but left the throw at line 46. This one is real, self-inflicted, and cheap — the strongest candidate in the set. Risk if skipped: any snapshot promote failure becomes an uncatchable fatal `Error`.
- **Finding #6 (flock return ignored) is a gap in REQ-047's new `FileLock` helper.** Valid, but the failure only manifests on filesystems where advisory `flock` returns false (some NFS/overlay) — rare on the documented dev/CI targets. Fixing it is a 2-line guard; low cost, low reachability. Don't over-scope it.
- **Findings #3 and #4 must be reconciled against REQ-050 and REQ-048 before any fix.** #3 (completeness flag never true under pure paratest) sits on REQ-050's completeness design; #4 (short-write passes the `=== false` check) sits on REQ-048's atomic-write design. Each is either a residual edge REQ-050/048 didn't close or a consequence they accepted. Capture must read those two archived REQs and decide, or a "fix" risks contradicting the accepted design (e.g. #3's premise that a single-process run self-heals may be the intended safety net).
- **Findings #7 and #8 are PLAUSIBLE-only with no reachable trigger today.** #7 needs all-weights-exactly-0.0 (the collector rounds real durations above 0.0; the no-timings path falls back to 1.0). #8 needs two roots in one process (the single caller passes an immutable root). Both are hardening-against-future-footgun, not live bugs — cheap defensive guards, but they should be scoped as such, not as urgent correctness.

## Connector — Links & Reuse

- **Finding #9 is the same duplication UR-011 left standing, and it couples to correctness.** `TestFileResolver::canonical()` vs `Support\Paths::canonical()` must produce byte-identical keys or timings degrade to count-balanced silently. Consolidating onto `Paths::canonical` (with the resolver keeping only its cache layer) removes the drift risk and the `warn()` / `--timings-dir` parse duplication in one altitude pass — but it touches recording *and* sharding key generation, so it needs its own tests, not a drive-by edit.
- **Footprint overlap will force serialization exactly as UR-011 hit.** `src/Timing/TimingStore.php` is touched by findings #3, #4, #9, #10; `src/Cli/ShardCommand.php` by #1, #5, #10. The UR-011 precedent (`2026-07-09 | UR-011 | ... TimingStore chain serialized 048→049→050→051→052 via hard deps`) applies again — capture should group by file/subsystem, not 1:1 finding→REQ, or the workers will footprint-conflict.
- **The test suite already covers this subsystem heavily** (TimingStoreTest 400 lines, TimingCollectorTest 212, ShardCommandTest 299, WarpBinTest 326). Each fix should extend the existing test file rather than add parallel ones, and #2's promote-failure path is currently untested — that gap is itself worth a REQ criterion.

## Summary

The single most important thing before decomposition: **finding #1 conflicts with the standing UR-011 read-only-overlay decision (REQ-051) — do not capture it as a straight "switch to committed-only" fix.** Either it is invalid (no reachable divergence in the intended flow) or it demands re-opening that decision at a question gate; capture must not silently reverse it. Findings #2 and #6 are genuine regressions/gaps in UR-011's own REQ-047 and are the safe, high-value fixes. Findings #3/#4 need reconciling against REQ-050/048's accepted designs before touching them, and #7/#8/#9/#10 are hardening/cleanup to be scoped as such — group everything by subsystem file to avoid the footprint serialization UR-011 already proved necessary.
