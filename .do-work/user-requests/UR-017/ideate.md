# Ideate — UR-017

**Reviewed:** 2026-07-10

## Explorer — Assumptions & Perspectives

- The brief's goal ("a /code-review re-run with no issues found") assumes a recall-biased high-effort review can return empty. It can't be guaranteed — that mode surfaces PLAUSIBLE-tier candidates by design. The workable acceptance bar (stated in the brief's context note) is zero **confirmed correctness** findings and no reportable cleanup findings; close should validate against that bar, not a literal empty array.
- Windows correctness (findings 10, 14 — drive-letter absolutization) is assumed in scope, but the project has no Windows CI and no declared Windows support; every prior UR tested on darwin/linux only. If Windows is best-effort, these fixes land as unit-tested code paths with no platform verification — fine, but the REQ acceptance criteria must say "unit-tested with simulated drive-letter inputs", not claim platform support.
- CI consumers restoring **read-only** timing artifacts are a first-class stakeholder per standing decision 2026-07-09 UR-011 ("read-only CI artifact restores must work"). Stream 2's "take the merge lock on the read path" affects exactly them — see Challenger.

## Challenger — Risks & Edge Cases

- **Direct contradiction with a standing decision.** 2026-07-10 UR-016: "canonical timing-key root ... mismatch **fails loudly** | user chose at question gate; getcwd keying silently degraded sharding." The brief's accepted decision for finding 7 is the exact reversal (degrade to count-balanced with warning). Both positions are defensible: fail-loudly protects against silently unbalanced sharding; degrade protects a CI matrix from a restored cache with a different workspace path. If degrade wins, capture must append a superseding decisions.md line and the warning must be unmissable; a middle path exists (degrade only when zero keys would match under the stored root, fail loudly otherwise).
- **Lock-on-read vs read-only load.** FileLock opens the lock file for writing; on a read-only restored artifact directory, acquiring merge.lock in `load()`/`fileTotals()`/`storedRoot()` would throw where today's lockless read succeeds — breaking the UR-011/UR-016 read-only guarantee to fix a race. The fix needs a fallback: attempt the lock, proceed lockless (with the current vanished-batch tolerance) when the lock cannot be created. Also: one lock acquisition must cover storedRoot+fileTotals together or the TOCTOU between them (cleanup finding 17) survives.
- **Errored-telemetry double-count.** Errored fires for prepared tests too (which also fire Finished). Recording `seconds()` from Errored unconditionally double-records prepared-but-failing tests. The fix must record via Errored only when the test never reached Finished (wasPrepared false), and telemetry seconds measure since-previous-event, not test duration — the REQ must pin down which duration is actually captured, with a test proving no double-record.
- **Root-scoped merge semantics are undefined.** "Reject or segregate batches whose root differs" leaves open: which root is authoritative (existing artifact's? first batch's?), and what happens to rejected batches — deleting loses data silently; leaving them pending means they're re-rejected and re-warned on every merge forever, and the pending dir grows unboundedly. Capture must specify: authoritative root = existing artifact root (or first batch when artifact absent), rejected batches warn-and-delete under the merge lock (consistent with UR-013's junk-batch precedent).
- **"Allow outside-root everywhere" changes the key domain.** Outside-root files currently get absolute, machine-specific keys (Paths::canonical allowOutside path) — enabling that in all three discovery modes spreads unstable keys further and reintroduces cross-machine mismatch, the exact failure class the root handshake exists to stop. Decide the key form for outside-root files (root-relative with ../ segments is stable across machines with the same layout) before widening the flag.
- **Fix-order coupling inside Stream 1.** Finding 4's fix (suiteRoot must realpath the implicitly discovered config, matching the standing UR-016 decision "root is the phpunit config file directory on both record and shard sides") changes what storedRoot comparisons see; landing it after the degrade-on-mismatch change hides regressions behind warnings. Sequence: shared root resolver first, then mismatch policy.

## Summary

The one thing that must be settled before capture: finding 7's accepted fix (degrade on root mismatch) reverses a user-chosen standing decision from yesterday ("mismatch fails loudly") — either confirm the reversal with a superseding decisions.md line, or adopt the middle path (fail loudly only when the stored root would still match some keys; degrade when the artifact is plainly from another machine). Second, the lock-on-read fix must preserve the UR-011 read-only-restore guarantee via a lockless fallback. Third, capture should group Stream 1 as one design REQ (UR-016 precedent) and keep the rest narrow (UR-015 precedent), with root-scoped-merge semantics pinned explicitly.
